# `Cargo.lock` y la reproducibilidad

## El problema de la reproducibilidad

«Compila en mi máquina y no en la tuya» es una de las frases más comunes en proyectos colaborativos. En el flujo de `L3`, el compilador no obligaba a fijar versiones de bibliotecas: si dos máquinas tenían distintas versiones de `libssl`, el binario podía construirse igual en ambas y comportarse distinto. Detectarlo era responsabilidad del proyecto —vía documentación, vía un `Makefile` que validaba versiones, vía contenedores que congelaban el entorno—, pero no había una pieza que el flujo de build proporcionara por defecto para resolverlo.

Cargo aborda este problema con un archivo separado del manifiesto: `Cargo.lock`. El manifiesto declara qué se necesita; el lockfile registra qué se eligió en un build dado.

## Manifiesto y lockfile son cosas distintas

El manifiesto declara **rangos** o **requisitos** de versión:

```toml
[dependencies]
anyhow = "1.0.102"
```

La cadena `"1.0.102"` admite cualquier versión `1.x.y` con `x.y >= 0.102` que sea compatible según semver. Si solo se mira el manifiesto, dos máquinas que corran `cargo build` en momentos distintos podrían resolver la dependencia a versiones distintas: la primera elige `1.0.102` porque es la última disponible cuando construye, la segunda elige `1.0.105` porque pasaron unas semanas y se publicaron parches. Las dos satisfacen el requisito.

El lockfile, en cambio, registra **versiones exactas** y huellas criptográficas:

```text
$ head -20 Cargo.lock
# This file is automatically @generated by Cargo.
# It is not intended for manual editing.
version = 4

[[package]]
name = "anyhow"
version = "1.0.102"
source = "registry+https://github.com/rust-lang/crates.io-index"
checksum = "0eb22ac35aedc91d8a7b5c2b18ddee2a30b9907ae3d6f72e83ee7db0ccd55d5d"

[[package]]
name = "saludo"
version = "0.1.0"
dependencies = [
 "anyhow",
]
```

Cada `[[package]]` registra el nombre, la versión exacta resuelta, la fuente (registry, path o git) y el `checksum` cuando viene de un registry. Si dos máquinas comparten el mismo `Cargo.lock`, ambas resuelven el grafo a las mismas versiones, descargan los mismos crates con los mismos checksums y se quejan si el contenido descargado no coincide con la huella registrada. La fijación es estricta y verificada.

## Cuándo se actualiza el lockfile

Cargo actualiza `Cargo.lock` solo en situaciones específicas:

- **Cuando no existe.** Si un build empieza sin lockfile, `cargo` resuelve el grafo desde el manifiesto y genera uno.
- **Cuando se agrega una dependencia.** `cargo add anyhow` agrega la línea al manifiesto y agrega la entrada correspondiente al lockfile.
- **Cuando se quita una dependencia.** El manifiesto pierde la línea, el lockfile pierde la entrada y todas sus dependencias transitivas que dejen de ser necesarias.
- **Cuando una dependencia nueva del manifiesto introduce un grafo que no se puede satisfacer con las versiones ya lockeadas.** En ese caso `cargo` reabre la resolución para las partes afectadas, manteniendo lo demás.
- **Cuando se invoca `cargo update`.** Este subcomando reabre la resolución completa o parcial y elige las versiones más recientes compatibles con los rangos del manifiesto.

`cargo build` por sí mismo **no** actualiza el lockfile si nada cambió en el manifiesto. Esto es importante: salvo que se ejecute `cargo update`, las versiones quedan fijas. En la práctica, eso significa que un build que funcionó en una máquina hoy seguirá funcionando mañana con el mismo lockfile, aunque `crates.io` haya publicado nuevas versiones de las dependencias.

`cargo update` sin argumentos reabre toda la resolución; con `cargo update -p <crate>` se reabre solo para una dependencia específica. Una corrida típica sin cambios disponibles imprime:

```text
$ cargo update
    Updating crates.io index
     Locking 0 packages to latest Rust 1.95.0 compatible version
note: pass `--verbose` to see 1 unchanged dependencies behind latest
```

Cero cambios. Si hubiera un patch nuevo compatible, aparecería una línea «Updating anyhow v1.0.102 -> v1.0.105» y el `Cargo.lock` se modificaría.

## Cuándo commitear `Cargo.lock` y cuándo no

La convención de la comunidad Rust, documentada en la guía oficial de cargo, es:

- **Para un binario o una aplicación: commitear `Cargo.lock`.** El lockfile garantiza que el binario que la persona usuaria recibe se construye igual que en CI y que en la máquina del autor.
- **Para una biblioteca publicada como crate: no commitear `Cargo.lock`.** Una biblioteca se publica para ser consumida por otros packages que tendrán su propio lockfile; el lockfile de la biblioteca no influye en quien la usa, y mantenerlo introduce ruido en los diffs.

Esta distinción se traduce en el `.gitignore` típico: para un binario se commitea `Cargo.lock`; para una biblioteca, se ignora. La regla práctica es mirar si el package es **el final** del grafo (binario, herramienta, servicio) o si se compone con otros (biblioteca pura).

Hay un caso ambiguo —un package que produce a la vez una biblioteca y un binario— donde la convención es commitear el lockfile, porque el binario lo necesita.

## Reproducir un build en otra máquina

La consecuencia operativa de tener `Cargo.lock` versionado es directa. Entregar `Cargo.toml` solo a otra persona hace que su build elija versiones potencialmente distintas; entregar `Cargo.toml` y `Cargo.lock` juntos hace que su build resuelva exactamente el mismo grafo que el original. El comando `cargo build --locked` exige que `cargo` use el lockfile sin modificarlo: si por algún motivo el manifiesto requiere reabrir la resolución, el comando falla en vez de actualizarlo silenciosamente. Es la flag que se suele usar en CI.

```text
$ cargo build --locked
   Compiling anyhow v1.0.102
   Compiling saludo v0.1.0 (/tmp/l4-lab/saludo)
    Finished `dev` profile [unoptimized + debuginfo] target(s) in 1.42s
```

Si se borra `Cargo.lock` antes y se corre el mismo comando, falla:

```text
$ rm Cargo.lock
$ cargo build --locked
error: the lock file /tmp/l4-lab/saludo/Cargo.lock needs to be updated but --locked was passed to prevent this
If you want to try to generate the lock file without accessing the network, remove the --locked flag and use --offline instead.
```

Esta es la diferencia material respecto al flujo de `L3`: el build system distingue, por sí mismo, entre «quiero reproducir lo que estaba lockeado» y «quiero reabrir la resolución». No depende de un script auxiliar ni de una convención del proyecto.

## Comparación con `L3`

El flujo `gcc + make` no tiene un equivalente directo de `Cargo.lock`. Si un `Makefile` invocaba `pkg-config --cflags --libs openssl`, las flags resultantes dependían de qué versión de `openssl-dev` estaba instalada en cada máquina. El binario producido podía linkar contra `libssl 1.1` en una máquina y `libssl 3.0` en otra, sin que ningún archivo del proyecto registrara cuál se eligió. Resolver esto requería sumar piezas externas: un contenedor que congelara las bibliotecas del sistema, un script que validara versiones de paquetes del distro, una sección del README que documentara dependencias.

`Cargo.lock` integra esa función al build system. La función no es magia: es un archivo generado, plano, leíble, que cualquiera puede inspeccionar. Pero está siempre, está estandarizado y lo respetan todos los subcomandos de cargo. La novedad de `L4` no es la reproducibilidad como concepto —los proyectos C grandes ya la perseguían—, es que el build system la entrega por defecto sin que el proyecto tenga que armarla a mano.

## Cierre

Después de este capítulo, frente a la pregunta «si te paso `Cargo.toml` solo, ¿podés reconstruir mi binario exactamente igual?», la respuesta tiene matices:

- con `Cargo.toml` solo, no: tu build podría elegir versiones más nuevas dentro de los rangos;
- con `Cargo.toml` y `Cargo.lock` y la flag `--locked`: sí, salvo que la toolchain difiera o que las dependencias hayan sido yanked del registry;
- en cualquier caso, el lockfile es el archivo que separa una resolución reproducible de una resolución oportunista.
