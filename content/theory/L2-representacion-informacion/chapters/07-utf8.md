# UTF-8 y codificación de longitud variable

## Code points y caracteres abstractos

ASCII alcanzó hasta donde llegaba el inglés. Para representar todos los sistemas de escritura del mundo —alfabetos europeos con acentos, alfabetos no latinos, ideogramas asiáticos, emojis, símbolos matemáticos— hace falta separar dos cosas que ASCII confundía: el caracter abstracto, como entidad simbólica, y la representación concreta de ese caracter como bytes.

La separación la hace el estándar **Unicode**, mantenido desde 1991 por el Unicode Consortium. Unicode asigna un número único —llamado **code point**— a cada caracter abstracto del mundo. Los code points se escriben con la notación `U+XXXX` donde los dígitos son hexadecimales. Algunos ejemplos:

- `U+0041` = `A` (letra latina mayúscula A)
- `U+00F1` = `ñ` (letra latina minúscula con tilde de eñe)
- `U+20AC` = `€` (signo del euro)
- `U+4E2D` = `中` (ideograma del chino mandarín que significa "centro")
- `U+1F600` = `😀` (emoji de cara sonriente)

El espacio total de code points abarca desde `U+0000` hasta `U+10FFFF`, lo que da poco más de un millón de puntos disponibles. La mayoría todavía no está asignada; las versiones más recientes del estándar usan alrededor de 150 000.

Los primeros 128 code points —`U+0000` hasta `U+007F`— están definidos para coincidir exactamente con la tabla ASCII. Esto no es casualidad: es una decisión del estándar para mantener compatibilidad con la enorme cantidad de texto e infraestructura preexistente. La letra `A`, que en ASCII era el byte `0x41`, en Unicode es `U+0041`. El número es el mismo.

Lo que el code point **no dice** es cómo se almacena en bytes. Esa parte la fija una **codificación de Unicode**: una regla que mapea code points a secuencias de bytes. Existen varias —UTF-16, UTF-32, UTF-8—, pero la dominante en la web, en archivos de texto, en el cómputo moderno, es **UTF-8**.

## Por qué un byte ya no alcanza

Un solo byte cubre 256 valores. Unicode tiene más de un millón de code points. La conclusión es directa: hace falta usar más de un byte para los code points fuera del rango ASCII.

El problema no es solo de tamaño, es de compatibilidad. UTF-8 fue diseñado en 1992 por Ken Thompson y Rob Pike con tres restricciones a la vez:

- **ASCII puro debe seguir funcionando bit a bit.** Los archivos ASCII existentes deben ser archivos UTF-8 válidos sin modificación.
- **El byte 0x00 no debe aparecer fuera del code point `U+0000`.** Las cadenas terminadas en byte cero al estilo C siguen funcionando, sin que un caracter no-ASCII inserte un cero en medio.
- **La codificación debe ser autosincrónica.** Mirando un byte aislado, sin ver los anteriores, debe poderse decir si ese byte es ASCII, el inicio de un caracter multibyte, o un byte de continuación. Esto permite que un programa que se posiciona en cualquier byte aleatorio pueda volver a alinearse con caracteres reales en pocos pasos.

UTF-8 cumple las tres simultáneamente. Las tres se manifiestan en el patrón de bits altos del byte, que es lo que viene a continuación.

## El esquema de bytes en UTF-8

Cada code point se codifica en 1, 2, 3 o 4 bytes según su rango. Las cuatro reglas son:

| Rango de code points | Bytes | Patrón en binario |
|---|---|---|
| `U+0000` – `U+007F` | 1 | `0xxxxxxx` |
| `U+0080` – `U+07FF` | 2 | `110xxxxx 10xxxxxx` |
| `U+0800` – `U+FFFF` | 3 | `1110xxxx 10xxxxxx 10xxxxxx` |
| `U+10000` – `U+10FFFF` | 4 | `11110xxx 10xxxxxx 10xxxxxx 10xxxxxx` |

Las `x` representan bits del code point, distribuidos de mayor a menor peso entre los bytes —el bit más significativo del code point va en la posición más alta disponible del primer byte—. Los bits que no son `x` son **fijos**: forman las cabeceras que identifican el rol del byte.

Las reglas de identificación, leídas a partir de los bits altos del byte:

- Si el byte empieza con `0`: es un caracter ASCII de 1 byte.
- Si empieza con `110`: es el primer byte de un caracter de 2 bytes.
- Si empieza con `1110`: es el primer byte de un caracter de 3 bytes.
- Si empieza con `11110`: es el primer byte de un caracter de 4 bytes.
- Si empieza con `10`: es un byte de **continuación**, parte de un caracter multibyte iniciado antes.

Esta es la propiedad de autosincronización: el patrón de bits altos de cualquier byte basta para clasificarlo. Si un programa se posiciona en un byte de continuación —`10xxxxxx`—, sabe que tiene que retroceder hasta encontrar un byte que empiece con `11` o con `0`, y desde ahí puede empezar a leer caracteres reales.

## Bits altos como marcadores de límite

La estructura de bits altos también explica cómo se reconstruye un code point a partir de su codificación. Para un caracter de 2 bytes `110xxxxx 10yyyyyy`, los bits del code point son los `x` y los `y` concatenados, con los `x` en la parte alta. Hay 5 bits `x` y 6 bits `y`, total 11 bits, que cubren el rango $[0, 2^{11} - 1] = [0, 2047]$. Como los code points de 2 bytes empiezan en `U+0080` y llegan hasta `U+07FF`, los 11 bits alcanzan justo.

Un ejemplo trabajado, sobre la letra `ñ` (`U+00F1`):

- Code point: `U+00F1`, que en binario es `0000 0000 1111 0001`. Los 11 bits significativos son `000 1111 0001`.
- Distribución: los 5 bits altos (`00011`) van en las `x` del primer byte; los 6 bits bajos (`110001`) van en las `y` del segundo byte.
- Primer byte: `110` + `00011` = `11000011` = `0xC3`.
- Segundo byte: `10` + `110001` = `10110001` = `0xB1`.
- Codificación final: `0xC3 0xB1`.

Verificación: el byte `0xC3` empieza con `110`, marca de 2 bytes; el `0xB1` empieza con `10`, byte de continuación. La estructura es coherente.

Para caracteres de 3 bytes el procedimiento es análogo. La letra `中` (`U+4E2D`):

- Code point: `U+4E2D`, binario `0100 1110 0010 1101`. Los 16 bits significativos son `0100 1110 0010 1101`.
- Distribución: 4 bits en el primero, 6 en el segundo, 6 en el tercero.
- Primer byte: `1110` + `0100` = `11100100` = `0xE4`.
- Segundo byte: `10` + `111000` = `10111000` = `0xB8`.
- Tercer byte: `10` + `101101` = `10101101` = `0xAD`.
- Codificación: `0xE4 0xB8 0xAD`.

## ASCII como subconjunto puro

Los caracteres ASCII —code points `U+0000` a `U+007F`— tienen un caso especial dentro del esquema: caben en un solo byte con el bit alto en cero. La codificación es **idéntica** a la tabla ASCII: la letra `A` es `0x41`, el dígito `5` es `0x35`, el espacio es `0x20`. Ningún byte de codificación ASCII puede empezar con los patrones reservados (`110`, `1110`, `11110`, `10`), porque todos arrancan con `0`.

La consecuencia es que un archivo ASCII puro es, sin convertir nada, un archivo UTF-8 válido. Los programas que solo hablan ASCII y los que hablan UTF-8 conviven sin necesidad de traducir nada mientras el contenido se mantenga en el rango ASCII. Esta es la propiedad que hizo que UTF-8 desplazara a las otras codificaciones de Unicode: no obligó a romper compatibilidad con el ecosistema preexistente.

## Caracteres no son bytes

Una consecuencia directa del esquema variable: en UTF-8, **contar bytes no es contar caracteres**. La cadena `"Hola"` ocupa 4 bytes (`H`, `o`, `l`, `a`, todos ASCII). La cadena `"Holá"` —con tilde en la `a` final— ocupa 5 bytes: `H` (0x48), `o` (0x6F), `l` (0x6C), `á` (0xC3 0xA1). La cadena tiene 4 caracteres pero 5 bytes.

La función `strlen` de C cuenta **bytes**, no caracteres, porque su contrato es "recorrer hasta encontrar el byte 0x00". Aplicada a `"Holá"` retorna 5. Para contar caracteres en UTF-8 hace falta una función específica que entienda el esquema de cabeceras —la lógica está disponible en bibliotecas como ICU o en las funciones nativas de lenguajes como Rust o Python 3—. Confundir bytes con caracteres es uno de los bugs más comunes en código que nació en entornos ASCII y se extendió a internacionalización: los buffers se calculan con la métrica equivocada y el texto se trunca, se duplica o se corrompe.

Otra consecuencia: el acceso aleatorio a una cadena UTF-8 no es directo. Para encontrar el caracter número $N$ hay que leer desde el principio, contando caracteres a medida que se reconocen las cabeceras. En ASCII puro, el caracter $N$ está en el byte $N$; en UTF-8 general, no. Los lenguajes que tratan cadenas como secuencias de caracteres —Python 3, por ejemplo— suelen pagar este costo internamente, almacenando las cadenas en una representación interna de ancho fijo o llevando índices auxiliares.

## Variabilidad como propiedad estructural, no exclusiva del texto

La idea de "patrón con bits altos que marcan rol y permiten longitud variable" no es exclusiva de UTF-8. Reaparece, en otra forma, en el **encoding de instrucciones máquina**.

En las arquitecturas x86 y x86-64 —las que dominan el cómputo de propósito general en computadoras de escritorio y servidores—, las instrucciones tienen longitud variable: ocupan 1, 2, 3, hasta 15 bytes en casos extremos. El primer byte —o el primer par de bytes en caso de prefijos especiales— actúa como cabecera y le dice al decodificador del procesador cuántos bytes siguen leyendo. Como en UTF-8, los bytes posteriores no tienen valor numérico autónomo: son operandos, modos de direccionamiento, datos inmediatos, fragmentos del opcode extendido. Y como en UTF-8, un byte aislado de un programa x86-64 puede ser parte del medio de una instrucción multi-byte, sin un valor independiente que tenga sentido.

Por contraste, las arquitecturas ARM y RISC-V, que aparecen más tarde en el track, usan instrucciones de **ancho fijo** —típicamente 4 bytes— que evitan la complicación del decodificador variable a cambio de un código un poco más voluminoso. Las dos estrategias —longitud variable con cabeceras, longitud fija sin cabeceras— resuelven el mismo problema con criterios opuestos, y reaparecen en distintos lugares del cómputo cada vez que hay que codificar entidades de tamaño heterogéneo en una secuencia de bytes.

La conclusión que importa para `L2`: cuando aparezca un dump de un archivo `.o` en `L3` o de un binario ejecutable más adelante, los bytes no se van a poder leer como caracteres —no son texto— ni como enteros sueltos —no son cifras—. Van a ser instrucciones codificadas con su propia gramática de longitud variable, que comparte con UTF-8 la idea estructural de cabeceras y continuaciones. La convención que se aplique va a depender de qué sección del archivo se esté mirando, y la pregunta operativa del nivel —patrón más convención produce valor— se va a aplicar exactamente igual.
